# Contributing to LingXi

感谢你帮助改进 LingXi。我们欢迎可复现的问题报告、聚焦的功能建议、文档修正和经过验证的代码贡献。

## 开始之前

1. 在目标仓库搜索已有 Issue 和 Pull Request，避免重复工作。
2. 对较大功能、公共 API 或架构调整，先创建 Issue 说明问题、约束和候选方案。
3. 安全漏洞不要创建公开 Issue，请遵循 [SECURITY.md](SECURITY.md)。
4. 参与项目即表示同意遵守 [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md)。

## 开发流程

1. Fork 目标仓库并从默认分支创建聚焦的功能分支。
2. 保持改动范围单一，不混入无关重构或格式化。
3. 为行为变化添加或更新测试，并同步相关文档、示例和锁文件。
4. 运行目标仓库 README 或贡献指南列出的格式、类型、测试与构建检查。
5. 提交 Pull Request，清楚说明动机、实现、影响、验证结果和兼容性风险。

## 提交与 Pull Request

- 使用简短、命令式的提交标题，例如 `feat: add revision validation`。
- 不要提交密钥、Token、个人数据、数据库备份、构建产物或本地环境文件。
- 如果更新依赖或 Git 子模块，请提交确定的锁文件或子模块 commit，并说明上游兼容性验证。
- 破坏性变更必须明确标注迁移方式和回滚方案。
- UI 变化应附截图；运行时变化应附最小复现或测试。

维护者可能要求缩小范围、补充测试、更新文档或重新整理提交。请保持讨论具体、友善并围绕技术事实。

## 报告问题

好的问题报告通常包含：

- 使用的仓库版本、commit、操作系统和运行环境；
- 最小复现步骤、预期行为与实际行为；
- 已脱敏的日志、堆栈或截图；
- 已尝试的排查方式；
- 对回归问题，给出最后正常版本。

请勿粘贴真实凭据或未经授权的数据。
